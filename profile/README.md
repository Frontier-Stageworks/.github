# Frontier Stageworks

**Professional filmmaking tools for people building outside the system.**

Frontier Stageworks builds practical technology for independent directors, small studios, animation teams, VFX shops, and filmmakers who are used to solving hard problems themselves.

We are interested in the space between expensive professional systems and improvised workarounds. That is where small teams are often doing the most inventive work, but it is also where the available tools tend to be closed, overpriced, fragile, or designed for productions with entirely different resources.

We want to help change that.

## A bigger scene, not a bigger gate

The independent music world of the 1990s did more than release records. Bands shared contacts. Labels published guides. People explained how to book shows, manufacture records, find distribution, and start something of their own.

The goal was not to become the only label that mattered.

The goal was to make it easier for more people to participate.

That is the spirit behind Frontier Stageworks.

We believe filmmaking technology should be more accessible, more understandable, and more open to modification. Small productions should be able to use professional techniques without needing enterprise budgets, proprietary ecosystems, or a dedicated engineering department.

Build the tool. Document the process. Share what can be shared. Help the next crew get started.

## What we build

Our projects focus on the practical problems that appear where filmmaking, animation, real-time graphics, camera tracking, motion control, physical production, and software meet.

That includes:

* Tools for independent and small-scale virtual production
* Camera tracking and real-time production systems
* Workflow utilities for animation and motion control
* Bridges between software, hardware, and production equipment
* Open-source projects that solve specific production problems
* Documentation that helps people understand, adapt, and build their own systems

Some projects will be open source. Some will become commercial products. We do not see those as opposing models.

Open-source software can strengthen the wider filmmaking community. Commercial products can support the sustained engineering, testing, documentation, manufacturing, and support required for dependable production tools.

In both cases, the goal is the same: make useful technology that respects the people using it.

## Projects

### [DragonMIDI](https://github.com/Frontier-Stageworks/dragonmidi)

DragonMIDI is an open-source application that brings physical MIDI control to Dragonframe.

It supports the KORG nanoKONTROL Studio and nanoKONTROL2, allowing their faders, knobs, and transport controls to operate Dragonframe functions including playback, frame capture, and motorized axes in the ARC motion-control workspace.

DragonMIDI was built for stop-motion animators, motion-control operators, and filmmakers who want tactile, programmable control without waiting for a manufacturer to support their exact workflow.

It is also an example of how we want to build open-source production software: with public design documentation, explicit behavioral specifications, automated testing, and honest documentation of limitations.

### Stagehand — Coming Soon

Stagehand is a compact, field-capable camera tracking system for virtual production.

It is being developed for independent filmmakers, small studios, previs teams, VFX artists, and productions that need reliable camera tracking without building their workflow around a large centralized tracking system.

## Our principles

### Small teams deserve serious tools

Independent does not mean amateur. Small crews deserve tools built for demanding creative and technical work.

### Open source makes the whole scene stronger

We release code, contribute upstream, and share documentation when doing so can help others build, adapt, repair, or learn.

Not every product will be fully open source, but open source is a real commitment here, not a marketing label.

### Trust should be earned

We use software assurance to define expected behavior, test important workflows and failure conditions, trace design decisions to implementation, and document limitations honestly.

We do not ask users to trust our tools on reputation alone. We want to provide evidence.

### Operators should understand their tools

Good systems should help users see what they are doing, recognize failures, and diagnose problems.

Abstraction is useful. Mystery is not.

### Interoperability matters

We prefer open protocols, documented interfaces, replaceable components, and workflows that work with the equipment people already own.

### Documentation is part of the product

A tool is more useful when people can understand it, configure it, troubleshoot it, and learn from it.

### Share the ladder

Release the code. Explain the design. Document the dead end. Answer the question you wish someone had answered for you.

The goal is not only to build successful tools. It is to help build a larger, more capable independent filmmaking community.

## Build something

Frontier Stageworks is for the people turning spare rooms into stages, writing glue code between incompatible systems, building camera rigs from available parts, learning unfamiliar software because the shot demands it, and refusing to accept that professional workflows must belong only to large productions.

Use the tools. Read the code. Report the bugs. Question the assumptions. Improve the documentation. Fork something and make it stranger.

Then show someone else how you did it.
